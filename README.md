# Video Chat Room

> Browser-based video room with realtime chat, participant presence, WebRTC media connections, device state handling, and responsive mobile UI.

## 🎯 Описание

**Video Chat Room** — веб-приложение для видеокомнат до 4 участников. Пользователь может создать комнату, пригласить других по ссылке, видеть список участников, общаться в чате, включать и выключать камеру/микрофон и оставаться в комнате даже при частичных проблемах с устройствами.

Проект реализует полный realtime flow:

- создание комнаты и вход по ссылке;
- уникальные никнеймы внутри комнаты;
- список участников с состояниями камеры и микрофона;
- WebRTC-видеосвязь между участниками;
- Socket.IO signaling для обмена `offer`, `answer` и ICE candidates;
- realtime-чат;
- graceful fallback при недоступной камере, микрофоне или сервере;
- мобильная версия с выдвижным чатом и фокусировкой видеоплитки;
- unit, integration и E2E-тесты.

Главная цель реализации — собрать не просто UI-макет, а рабочую комнату с реальным обменом медиа между браузерами, устойчивой синхронизацией участников и понятной обработкой пограничных состояний.


### Клиент

Клиент отвечает за:

- маршрутизацию между стартовым экраном и комнатой;
- захват камеры и микрофона через `getUserMedia`;
- управление локальным состоянием медиа;
- установку WebRTC peer connections;
- отображение видеоплиток, участников и чата;
- мобильную адаптацию интерфейса.

Ключевые модули:

| Файл | Назначение |
| --- | --- |
| `client/src/app/App.jsx` | Основная композиция UI, комната, чат, видеоплитки, мобильный drawer. |
| `client/src/hooks/useRoom.js` | Socket.IO-подключение, вход в комнату, сообщения, список участников. |
| `client/src/hooks/useLocalMedia.js` | Работа с камерой/микрофоном и частичными ошибками устройств. |
| `client/src/hooks/usePeerConnections.js` | WebRTC peer connections, signaling, синхронизация треков. |
| `client/src/lib/media.js` | Утилиты для статуса медиа и сообщений об ошибках устройств. |
| `client/src/lib/videoTile.js` | Логика отображения видеоплитки и fallback-состояний. |

### Сервер

Сервер отвечает за:

- HTTP healthcheck;
- CORS для разрешённых client origins;
- Socket.IO lifecycle комнаты;
- хранение комнат и участников в памяти;
- relay WebRTC signaling-сообщений между участниками;
- чат и системные сообщения.

Ключевые модули:

| Файл | Назначение |
| --- | --- |
| `server/src/index.js` | Инициализация Express, HTTP server, Socket.IO и env-конфигурации. |
| `server/src/socketHandlers.js` | Socket.IO события комнаты, чата, media state и WebRTC signaling. |
| `server/src/roomStore.js` | In-memory модель комнат, участников, сообщений и бизнес-валидаций. |
| `server/src/validators.js` | Валидация room id, никнеймов и сообщений. |

## 💻 Использование

### Создание комнаты

1. Открыть `http://localhost:5173`.
2. Ввести никнейм.
3. Нажать `Создать комнату`.
4. Скопировать ссылку приглашения.

### Вход второго участника

1. Открыть ссылку комнаты во второй вкладке, другом браузере или на другом устройстве.
2. Ввести уникальный никнейм.
3. Нажать `Войти`.

### Управление звонком

| Действие | Поведение |
| --- | --- |
| Выключить микрофон | Другие участники видят muted-индикатор. |
| Включить микрофон | Audio track снова передаётся участникам. |
| Выключить камеру | Видео заменяется fallback-плиткой `камера выключена`. |
| Включить камеру | Видео восстанавливается без пересоздания комнаты. |
| Открыть чат на мобильном | Боковая панель выезжает поверх видео. |
| Нажать на видеоплитку на мобильном | Плитка увеличивается; повторное нажатие возвращает сетку. |

## ⚙️ Конфигурация

Проект читает переменные окружения из `.env` в корне репозитория.

| Переменная | Значение по умолчанию | Описание |
| --- | --- | --- |
| `PORT` | `3000` | Порт HTTP/Socket.IO сервера. |
| `CLIENT_ORIGIN` | `http://localhost:5173` | Разрешённые origins клиента для CORS. Несколько значений можно указать через запятую. |
| `VITE_SOCKET_URL` | `http://localhost:3000` | URL Socket.IO сервера для клиента. |
| `VITE_STUN_URLS` | `stun:stun.l.google.com:19302` | STUN-серверы для WebRTC, через запятую. |
| `STUN_URLS` | `stun:stun.l.google.com:19302` | Зарезервировано для серверной конфигурации STUN. |

Пример:

```env
PORT=3000
CLIENT_ORIGIN=http://localhost:5173
VITE_SOCKET_URL=http://localhost:3000
VITE_STUN_URLS=stun:stun.l.google.com:19302
STUN_URLS=stun:stun.l.google.com:19302
```

### Запуск в локальной сети

Для проверки с другого устройства нужно:

1. Указать IP машины с сервером и клиентом.
2. Разрешить origin клиента в `CLIENT_ORIGIN`.
3. Указать `VITE_SOCKET_URL` на сервер по локальному IP.

Пример:

```env
CLIENT_ORIGIN=http://192.168.3.12:5173
VITE_SOCKET_URL=http://192.168.3.12:3000
```

Важно: доступ к камере и микрофону в браузерах обычно требует защищённый контекст. Для другого устройства в локальной сети может понадобиться HTTPS, иначе браузер может показать, что WebRTC или media devices недоступны.

## 🧩 Самые сложные решённые проблемы

### 1. Стабильное восстановление камеры после выключения

Проблема: после выключения и повторного включения камеры у второго участника иногда оставался чёрный квадрат. При этом выключение микрофона внезапно “оживляло” видео.

Причина была в двух местах:

- видеотрек сначала останавливался и удалялся, из-за чего WebRTC sender/receiver мог оставаться в неудачном состоянии;
- React-эффект, который привязывает `MediaStream` к `<video>`, не срабатывал при возврате видеоэлемента, но случайно срабатывал при изменении состояния микрофона.

Решение:

- камера выключается через `track.enabled = false`, а не через уничтожение track;
- WebRTC sender остаётся стабильным;
- эффект привязки `srcObject` зависит от `hasLiveVideo`, поэтому видео корректно восстанавливается без переключения микрофона.

### 2. Старые участники не видели новых

Проблема: новый участник видел старого, но старый участник не всегда видел нового.

Причина: список участников обновлялся корректно, но WebRTC peer connections не строились надёжно от этого списка. Часть логики зависела от события `participant:joined`, которое могло прийти в неудачный момент. Кроме того, при создании соединений важно не создавать конкурирующие SDP offer с обеих сторон.

Решение:

- `participants:list` стал источником истины для сверки peer connections;
- недостающие `RTCPeerConnection` создаются при обновлении списка участников;
- offer создаёт только одна сторона по стабильному правилу `shouldInitiateConnection(localId, remoteId)`;
- локальные tracks добавляются до SDP negotiation только на нужной стороне.

### 3. Streamless `ontrack` в WebRTC

Проблема: в некоторых сценариях браузер отдавал `event.track`, но не отдавал готовый `event.streams[0]`. Старый код считал это отсутствием видео.

Решение:

- если `event.streams[0]` есть, используется он;
- если есть только `event.track`, клиент создаёт `MediaStream` вручную и добавляет track туда;
- это покрыто unit-тестами.

### 4. Частичная недоступность камеры или микрофона

Проблема: ошибка устройства была общей: приложение не понимало, что именно недоступно — камера или микрофон.

Решение:

- аудио и видео запрашиваются отдельно;
- состояние хранит `audioError` и `videoError`;
- если недоступна камера, блокируется только кнопка камеры;
- если недоступен микрофон, блокируется только кнопка микрофона;
- пользователь всё равно может находиться в комнате и пользоваться чатом.

### 5. Мобильная раскладка

Проблема: десктопная двухколоночная сетка плохо переносилась на маленький экран.

Решение:

- убран жёсткий `min-width` для мобильных экранов;
- видео участников отображается квадратной сеткой;
- чат и список участников вынесены в drawer;
- видеоплитку можно увеличить нажатием;
- добавлен E2E-тест мобильного drawer и фокуса плитки.

## 📌 Ограничения и особенности

- Комнаты и сообщения хранятся в памяти сервера, без базы данных.
- Mesh WebRTC подходит для малого количества участников; поэтому установлен лимит 4 человека.
- TURN-сервер не настроен. В сложных NAT-сетях peer-to-peer соединение может не установиться без TURN.
- Для доступа к камере и микрофону на другом устройстве может потребоваться HTTPS.
- Скриншоты интерфейса пока не добавлены.
